  <action>
Create `backend/scripts/migrate_firestore_to_postgres.py` as a standalone CLI script.

**CLI interface:**
```
python3 migrate_firestore_to_postgres.py --service-account /path/to/key.json --database-url postgresql://...
```
- `--service-account` (required): Path to Firestore service account JSON key
- `--database-url` (required): PostgreSQL connection URL (synchronous, psycopg2 format: `postgresql://user:pass@host:5432/db`)
- `--dry-run` (optional): Report counts without writing
- `--collections` (optional): Comma-separated list of collections to migrate (default: all)

**Script structure:**
1. Parse CLI args with `argparse`
2. Initialize Firestore client with service account JSON: `google.cloud.firestore.Client.from_service_account_json(path)`
3. Initialize SQLAlchemy sync engine from database URL (use psycopg2, not asyncpg)
4. Create all tables if not exist (`Base.metadata.create_all(engine)`)
5. For each collection, read all documents and insert into PostgreSQL

**Collection migration functions (one per collection):**

Each function:
- Reads all docs from Firestore collection
- Maps Firestore document fields to SQLAlchemy model columns
- Uses `session.bulk_save_objects()` for batch insert
- Returns (firestore_count, postgres_count) tuple

**Collection mapping details:**

| Collection | Model | Key field mapping notes |
|------------|-------|------------------------|
| `users` | User | Firestore doc ID -> User.id, skip password_hash (users will re-register) |
| `jobs` | Job | Firestore doc ID -> Job.id, tool field needs ToolType enum mapping |
| `extract_entries` | ExtractEntry | job_id from parent ref or field |
| `title_entries` | TitleEntry | job_id from parent ref or field |
| `proration_rows` | ProrationRow | job_id from parent ref or field |
| `revenue_statements` | RevenueStatement | Nested `rows` array -> separate RevenueRow inserts |
| `rrc_oil_proration` | RRCOilProration | Large collection -- batch 500 at a time |
| `rrc_gas_proration` | RRCGasProration | Large collection -- batch 500 at a time |
| `rrc_data_syncs` | RRCDataSync | Direct field mapping |
| `rrc_county_status` | RRCCountyStatus | Doc ID -> key field |
| `audit_logs` | AuditLog | Direct field mapping |
| `app_config` | AppConfig | Doc ID -> key, remaining fields -> data JSONB |
| `user_preferences` | UserPreference | Doc ID or user_id field |
| `ghl_connections` | GHLConnection | Doc ID -> id |
| `rrc_sync_jobs` | RRCSyncJob | Doc ID -> id |
| `rrc_metadata` | RRCMetadata | Doc ID -> key |

**Error handling:**
- Wrap each collection migration in try/except -- log error and continue to next collection
- On duplicate key (already migrated), skip with warning
- Print summary table at end: collection name, Firestore count, PostgreSQL count, status (OK/SKIPPED/ERROR)

**Important:** This script uses `firebase-admin` and `google-cloud-firestore` for reading only. These packages are NOT in requirements.txt (removed in Plan 01). The script should have a comment at the top explaining: "Run with: pip install firebase-admin google-cloud-firestore" -- these are one-time dependencies for migration only.

**Batch strategy for large collections (RRC data):**
- Read in batches of 500 using Firestore `.limit(500).offset()` or stream()
- Insert in batches of 500 using `session.bulk_save_objects(batch); session.commit()`
- Print progress every 1000 records

**Verification output:**

```
jobs                  |      142  |       142  | OK
...
```
