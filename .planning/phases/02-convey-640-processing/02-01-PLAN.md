  <behavior>
    ## Schema Validation (CSV-01)
    - parse_convey640(csv_bytes, "test.csv") with valid 12-column CSV returns Convey640ParseResult with entries
    - parse_convey640(xlsx_bytes, "test.xlsx") with valid 12-column Excel returns Convey640ParseResult with entries
    - parse_convey640(csv_bytes, "test.csv") with missing columns raises ValueError listing missing column names
    - parse_convey640(empty_csv, "test.csv") raises ValueError (no data rows)

    ## Entry Number Stripping (CSV-02)
    - Name "104 INA NADINE TAYLOR" -> entry_number="104", primary_name starts with "INA NADINE TAYLOR"
    - Name "AARON TRACY" (no number) -> entry_number is row-index-based, primary_name="AARON TRACY"
    - Name "2WOOD OIL & GAS LLC" -> entry_number is row-index-based, primary_name="2WOOD OIL & GAS LLC" (no space after digit, not an entry number)

    ## Name Normalization (CSV-02)
    - "JOHN DOE DECEASED" -> primary_name contains "JOHN DOE", entity_type=ESTATE, notes contains "deceased"
    - "JAMES E DESHIELDS JR & RITA F DESHIELDS" -> primary_name for first person, notes contains "RITA F DESHIELDS"
    - "INA NADINE TAYLOR REVOCABLE TRUST DATED..." -> entity_type=TRUST, primary_name extracted as grantor "INA NADINE TAYLOR", trust details in notes
    - "ALISHA BRUMMETT NOW RECKER" -> primary_name contains "RECKER" (married name), notes contains maiden name
    - "ROSE MAURICE PODDER NEE YOUNGHEIM" -> primary_name contains "PODDER", notes contains "f/k/a YOUNGHEIM"
    - "JOHN YOUNGHEIM DECEASED CLO LYNNE YOFFE" -> entity_type=ESTATE, notes contains "c/o LYNNE YOFFE"
    - "JUDI K SMITH AS TRUSTEE OF THE JUDI K SMITH TRUST" -> entity_type=TRUST, primary_name="JUDI K SMITH"
    - Entity detection: "2WOOD OIL & GAS LLC" -> entity_type=LLC (do NOT split on &)

    ## ZIP Code Preservation (CSV-03)
    - postal_code "73071.0" -> zip_code="73071"
    - postal_code "2668.0" -> zip_code="02668" (leading zero preserved)
    - postal_code "nan" -> zip_code="" (empty string)
    - postal_code "783160.0" -> zip_code="78316" (truncated, entry flagged)

    ## Metadata Extraction (CSV-04)
    - county column -> CaseMetadata.county
    - str column -> CaseMetadata.legal_description
    - applicant column -> CaseMetadata.applicant
    - case_no "2026000909" -> CaseMetadata.case_number="CD 2026-000909-T"
    - curative "0" -> section_type="regular", curative "1" -> section_type="curative"

    ## Curative/Section Type Mapping
    - Row with curative="0" -> PartyEntry.section_type="regular"
    - Row with curative="1" -> PartyEntry.section_type="curative"
  </behavior>
  <implementation>
    Create `backend/app/services/extract/convey640_parser.py` as a single module following the ECF parser pattern:

    **Public API:**
    - `Convey640ParseResult` dataclass (entries: list[PartyEntry], metadata: CaseMetadata)
    - `parse_convey640(file_bytes: bytes, filename: str) -> Convey640ParseResult`

    **Internal pipeline (in order):**
    1. Read file with `dtype=str` and `keep_default_na=False` (CSV) or `dtype=str` (Excel)
    2. Validate schema against EXPECTED_COLUMNS set (12 columns)
    3. Extract metadata from first row via `_extract_metadata(df)`
    4. For each row, run name normalization pipeline:
       a. `_strip_entry_number(name)` -- regex `^\d+\.?\s+`, returns (entry_number|None, cleaned_name)
       b. `_extract_care_of(name)` -- CLO/ELO patterns, returns (cleaned_name, care_of|None)
       c. Also handle standard `C/O` pattern
       d. `_extract_aka(name)` -- A/K/A aliases to notes
       e. `_extract_nee(name)` -- NEE maiden name to notes
       f. `_extract_now_name(name)` -- NOW married name becomes primary, maiden to notes
       g. `_strip_deceased(name)` -- remove DECEASED/POSSIBLY DECEASED, flag entity_type=ESTATE
       h. Run `detect_entity_type()` from utils/patterns.py
       i. For individuals: split joint names on `&` (first person primary, rest to notes)
       j. For trusts: extract grantor name as primary, trust details to notes
       k. Run `parse_name()` for first/middle/last/suffix on individuals
    5. Build PartyEntry with normalized postal code, section_type from curative column
    6. Flag entries with anomalous data (6-digit ZIP, embedded address in name)

    **Key patterns to reuse:**
    - `detect_entity_type()` from `utils/patterns.py` for entity classification
    - `parse_name()` from `services/extract/name_parser.py` for first/middle/last splitting
    - `CaseMetadata` and `PartyEntry` models from `models/extract.py` (return identical types to ECF parser)

    **Per user decisions:**
    - Joint names: first person is primary, additional names go to notes
    - Trust names: extract grantor's personal name as primary (contactable person), trustee to notes
    - Entries without addresses: keep them, don't filter
    - Embedded-address anomalies: flag, don't split
    - Case number format: 'CD {year}-{number}-T'
    - curative 0->regular, 1->curative
  </implementation>
</feature>
