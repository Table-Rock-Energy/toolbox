## Phase Details

### Phase 9: Backend Foundation
**Goal**: Backend can securely store GHL connections, validate credentials, and communicate with GoHighLevel API
**Depends on**: Phase 8
**Requirements**: CONF-01, CONF-02, CONF-03, CONF-04
**Success Criteria** (what must be TRUE):
  1. Backend can store encrypted Private Integration Tokens in Firestore (never sent to frontend)
  2. Backend can validate token + Location ID together via test API call to GHL
  3. Backend can authenticate with GHL API and fetch user list for contact owner dropdown
  4. Backend can upsert a single contact with proper field normalization (phone to E.164, email trimmed/lowercased)
  5. Backend respects rate limits with throttling (100 req/10s burst limit) and exponential backoff for 429 responses
**Plans**: 2 plans

Plans:
- [ ] 09-01-PLAN.md — GHL Pydantic models, API client with rate limiting, contact normalization
- [ ] 09-02-PLAN.md — Connection CRUD service, GHL API router, main.py wiring

### Phase 11: Bulk Send Engine
**Goal**: System can reliably send contact batches to GHL with validation, formatting, and tagging
**Depends on**: Phase 9, Phase 10
**Requirements**: CTCT-01, CTCT-02, CTCT-03, CTCT-05, CTCT-06
**Success Criteria** (what must be TRUE):
  1. System validates required fields (email or phone) before sending batch and shows clear error if validation fails
  2. System upserts contacts to GHL (creates new contacts or merges new info into existing)
  3. System applies campaign tag from modal to all contacts in batch
  4. System applies "manual SMS" tag to contacts when checkbox is checked in modal
  5. System processes contacts in batches with rate limit awareness (50-80 contacts per batch)
  6. System tracks per-contact success/failure status with stable identifiers during send operation
**Plans**: 2 plans

Plans:
- [ ] 11-01-PLAN.md — Backend bulk send engine (models, service, API endpoints)
- [ ] 11-02-PLAN.md — Frontend send modal wiring (validate, confirm, send, results)

### Phase 12: Progress & Error Handling
**Goal**: Users see real-time progress and can diagnose failures with detailed error reports
**Depends on**: Phase 11
**Requirements**: SEND-03, SEND-04, SEND-05
**Success Criteria** (what must be TRUE):
  1. User sees real-time progress bar showing contact count (X of Y processed) via Server-Sent Events
  2. User sees running totals of created, updated, and failed contacts during send
  3. User sees summary modal after send completion with final created/updated/failed counts
  4. User sees specific error message for each failed contact in summary modal
  5. User can download a CSV of failed contacts with error details column
  6. Send operation runs as async job (POST returns job_id immediately, processing happens in background)
**Plans**: 2 plans

Plans:
- [ ] 12-01-PLAN.md — Backend async job, SSE progress streaming, cancel, error categorization
- [ ] 12-02-PLAN.md — Frontend SSE hook, progress/summary modal, failed contacts management + retry

## Notes

**v1.2 Phase Ordering Rationale:**
- **Parallel foundations (Phases 9 + 10):** Backend API client and frontend UI can be built simultaneously with mock data, accelerating delivery
- **Backend first priority:** Phase 9 includes encryption, token validation, and single contact upsert — all must be solid before bulk operations
- **Frontend can start immediately:** Phase 10 builds Settings UI and Send modal with stub data while Phase 9 implements real backend
- **Bulk send after foundations:** Phase 11 combines backend API (Phase 9) with frontend UI (Phase 10) for batch processing
- **Progress + errors combined:** Phase 12 merges SSE progress streaming with error handling (originally separate phases) for faster delivery
- **Polish last:** Phase 13 handles edge cases after core functionality is proven

**Dependency chain:** Phase 9 (backend API client) + Phase 10 (frontend UI, parallel) → Phase 11 (bulk send, requires both) → Phase 12 (progress + errors) → Phase 13 (production polish)

**Parallelization strategy:** Phase 9 and Phase 10 can run in parallel. Frontend developers build Settings UI and Send modal with hardcoded stub data (mock GHL connections, mock user dropdown) while backend developers implement encryption, token validation, and GHL API client. Once Phase 9 completes, Phase 10 frontend wires to real API endpoints. This cuts 1-2 weeks from critical path.

**Research Flags:**
- v1.2: No research needed — all patterns exist in codebase or well-documented in GHL API docs
