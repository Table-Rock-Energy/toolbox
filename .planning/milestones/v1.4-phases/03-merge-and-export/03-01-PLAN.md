<feature>
  <name>Merge Service: Entry-number matching with PDF precedence</name>
  <files>backend/app/services/extract/merge_service.py, backend/tests/test_merge_service.py</files>
  <behavior>
    ## MRG-01: PDF wins contact fields
    - Given PDF entry with name="JOHN SMITH" and CSV entry with name="J0HN SMITH" (OCR error), merged entry has name="JOHN SMITH"
    - Given PDF entry with address="123 Main St" and CSV entry with address="123 Main St", merged entry has address="123 Main St" (PDF value)
    - Given PDF entry with all contact fields populated, CSV contact fields are ignored entirely

    ## MRG-02: CSV metadata enriches merged result
    - Given PDF metadata with county=None and CSV metadata with county="CADDO", merged metadata has county="CADDO"
    - Given PDF metadata with county="CADDO" and CSV metadata with county="GRADY", merged metadata has county="CADDO" (PDF wins when both present)
    - Given CSV metadata with case_number and applicant, those fill gaps in PDF metadata
    - well_name comes only from PDF (CSV doesn't have it)

    ## MRG-03: Entry-number matching
    - Given PDF entries ["1","2","3"] and CSV entries ["1","2","3"], all 3 match
    - Given PDF entries ["1","2","3"] and CSV entries ["1","3"], entries 1 and 3 match, entry 2 is PDF-only (not flagged)
    - Matching uses exact string comparison of entry_number field

    ## MRG-04: Mismatch warnings
    - CSV-only entries (in CSV but not PDF) are included with flagged=True and flag_reason containing "No PDF match"
    - merge_result.warnings includes count summary: "X of Y entries matched"
    - When >50% of PDF entries are unmatched, warnings include fallback message and per-entry merge is skipped
    - In fallback mode, CSV metadata still populates export-level metadata fields

    ## Edge cases
    - csv_result=None returns PDF entries unchanged with PDF metadata, no warnings
    - Empty PDF entries list returns empty result
    - CSV entries with genuinely blank PDF fields (e.g., PDF has no address, CSV has address): CSV fills the blank
    - Existing PDF entry notes (e.g., "c/o Jane Smith") are preserved, not overwritten
  </behavior>
  <implementation>
    Create `backend/app/services/extract/merge_service.py` with:

    1. `MergeResult` dataclass with fields: `entries: list[PartyEntry]`, `metadata: CaseMetadata`, `warnings: list[str]`

    2. `merge_entries(pdf_result: ECFParseResult, csv_result: Convey640ParseResult | None) -> MergeResult` pure function:
       - If csv_result is None, return MergeResult with pdf entries, pdf metadata, empty warnings
       - Build dict: `csv_by_entry = {e.entry_number: e for e in csv_result.entries}`
       - Iterate pdf entries, look up matching csv entry via `csv_by_entry.pop(entry_number, None)`
       - For matched entries: call `_fill_blanks(pdf_entry, csv_entry)` to fill genuinely blank PDF fields from CSV
       - Track match count
       - Compute match_rate = matched / len(pdf_entries)
       - If match_rate < 0.5: fall back to PDF-only mode (skip per-entry merge, keep PDF entries unchanged, but still merge metadata)
       - Remaining csv_by_entry values are CSV-only entries: include with `flagged=True`, `flag_reason="No PDF match -- data from Convey 640 only (may contain OCR errors)"`
       - Merge metadata via `_merge_metadata(pdf_meta, csv_meta)`: CSV fills None fields in PDF metadata, except well_name (PDF only)
       - Build warnings list with match summary

    3. `_fill_blanks(pdf_entry: PartyEntry, csv_entry: PartyEntry) -> PartyEntry`:
       - For fields: mailing_address, city, state, zip_code
       - If PDF field is None/empty and CSV field has value, use `pdf_entry.model_copy(update={...})`
       - Never overwrite existing PDF values

    4. `_merge_metadata(pdf_meta: CaseMetadata, csv_meta: CaseMetadata | None) -> CaseMetadata`:
       - For county, legal_description, applicant, case_number: use `pdf_value or csv_value`
       - well_name: always PDF (CSV doesn't have it)
  </implementation>
</feature>
