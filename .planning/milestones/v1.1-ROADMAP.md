### Phase 1: Foundation & Quality Baseline
**Goal**: Developers can add new tools by implementing a simple Protocol interface, and the platform has comprehensive test coverage protecting against regressions.

**Depends on**: Nothing (foundation phase)

**Requirements**: FNDTN-01, FNDTN-02, FNDTN-03, FNDTN-04, FNDTN-05, FNDTN-06, DEVOPS-01, DEVOPS-02, DEVOPS-03, DEVOPS-04, DEVOPS-05, MIGRATE-01

**Success Criteria** (what must be TRUE):
  1. Developer creates new tool by implementing ToolProtocol (name, process(), get_input_schema(), get_output_schema()) without inheritance
  2. Developer registers tool via @register_tool decorator, tool appears in ToolRegistry automatically at startup
  3. Developer POSTs to /api/tools/{name}/process with any tool name, request validates against tool's input schema
  4. Platform validates tool output against declared output schema before returning to client
  5. User uploads invalid file to Extract tool via /api/tools/extract/process, sees structured error message without app crash
  6. Extract tool processes OCC Exhibit A PDF via new generic endpoint, returns same PartyEntry results as legacy /api/extract/upload
  7. CI pipeline runs pytest suite on every PR, fails merge if tests fail
  8. Developer runs "make test" locally, sees comprehensive backend and frontend test output

**Plans:** 5 plans

Plans:
- [ ] 01-01-PLAN.md — Plugin system core (ToolProtocol, ToolRegistry, base schemas, generic endpoint)
- [ ] 01-02-PLAN.md — CI pipeline setup and codebase audit
- [ ] 01-03-PLAN.md — Extract tool migration to plugin system
- [ ] 01-04-PLAN.md — Backend test suite (pytest for plugin system + generic endpoint)
- [ ] 01-05-PLAN.md — Frontend test setup (Vitest) and CI pipeline finalization

### Phase 2: Frontend Manifest System
**Goal**: New tools appear in UI navigation and routing automatically when registered, without modifying App.tsx or Sidebar.tsx.

**Depends on**: Phase 1 (requires ToolRegistry from backend)

**Requirements**: FE-01, FE-02, FE-03, FE-04, FE-05

**Success Criteria** (what must be TRUE):
  1. Developer registers tool in TypeScript ToolRegistry with displayName, route, icon, capabilities, input/output schemas
  2. User navigates to app, sees new tool in sidebar navigation without Sidebar.tsx modification
  3. User clicks tool in sidebar, route loads GenericToolPage component rendering file upload, data table, and export buttons from manifest
  4. Extract tool renders via GenericToolPage, user uploads PDF and sees party extraction results in data table
  5. GenericToolPage shows "Enrich Data" button only for tools where capabilities include enrichment

### Phase 3: Enrichment Pipeline
**Goal**: Users can enrich extracted data with address validation and AI cleanup on-demand, with pre-flight cost estimation and result caching.

**Depends on**: Phase 2 (requires GenericToolPage to display enrichment panel)

**Requirements**: ENRICH-01, ENRICH-02, ENRICH-03

**Success Criteria** (what must be TRUE):
  1. User processes data with Extract tool, sees "Enrich Data" button on results screen
  2. User clicks "Enrich Data", sees modal showing estimated cost ($X for Y addresses, Z AI validations) before running
  3. User approves enrichment, system deduplicates addresses before calling APIs, reduces API calls by 60-90%
  4. User enriches same dataset twice, second run is instant (cached results keyed by input hash)
  5. Title tool user enriches owner data with address validation, sees enriched addresses in data table
  6. Admin views enrichment cost dashboard showing per-tool, per-service spending history

### Phase 4: File & Web Scraper Adapters
**Goal**: Developers can add file upload sources and web scraping sources using standardized adapter interfaces, with built-in health monitoring.

**Depends on**: Phase 1 (requires SourceAdapterRegistry)

**Requirements**: SRC-01, SRC-02, SRC-03, SRC-04, SRC-05, SRC-06, SRC-07

**Success Criteria** (what must be TRUE):
  1. Extract tool uses FileUploadAdapter wrapping existing ingestion engine, processes PDFs via standardized adapter interface
  2. Developer creates web scraper tool by implementing ScraperProtocol, registers with @register_scraper decorator
  3. User navigates to scraper tool, tool uses Playwright for authenticated site (handles login, cookies, JS rendering)
  4. User navigates to simple scraper tool, tool uses httpx for public HTML-only site (lighter weight)
  5. Scraper runs nightly, detects page structure change (via hash), sends alert to admin
  6. User views scraper health dashboard showing success rate (70% last 7 days), last successful run timestamp
  7. User sees "Data last updated: 3 days ago" warning when scraper is stale
  8. Scraper fails for 5 consecutive runs, user uploads CSV manually as fallback

### Phase 5: API Source Adapter
**Goal**: Developers can integrate REST APIs as data sources with standardized auth, pagination, and rate limiting.

**Depends on**: Phase 4 (shares rate limiting and retry patterns)

**Requirements**: SRC-08, SRC-09

**Success Criteria** (what must be TRUE):
  1. Developer creates API adapter with OAuth2 authentication, adapter handles token refresh automatically
  2. Developer creates API adapter with API key auth, key read from environment variable securely
  3. API returns 10K records across 100 pages, adapter handles pagination automatically and yields all records
  4. API enforces 100 req/min rate limit, adapter respects limit and spaces requests accordingly
  5. API returns 429 rate limit error, adapter backs off exponentially and retries successfully

## Notes

**Phase ordering rationale:**
- Foundation first, features second (Protocol/registry validates with Extract before building enrichment or scrapers)
- Frontend manifest parallels enrichment work (Phase 2 and 3 could run in parallel if needed)
- Cost controls precede enrichment launch (caching + deduplication + cost dashboard in Phase 3 before user access)
- One-tool-at-a-time migration (Phase 1 migrates Extract, Phase 7 migrates remaining three with production validation between each)
- Monitoring built into new source types (Scraper and API phases include health dashboards from day one)
- **v1.1 GHL Prep independent** (Phase 8 delivers immediate value without dependency on v1.0 modular refactor)

**Research flags:**
- Phase 4 (Web Scraper): Playwright memory usage on Cloud Run (1Gi limit), browser authentication persistence patterns
- Phase 3 (Enrichment): Specific enrichment service APIs (pricing, rate limits, authentication) need research per type
- **Phase 8 (GHL Prep)**: No research needed — follows established CSV processing patterns from Title tool, zero new dependencies

**Migration strategy:**
- Strangler fig pattern: new generic endpoints coexist with legacy routes
- Feature flags enable toggling between implementations
- Production validation after each tool migration before proceeding
- <5 minute rollback plan for each migration
