# Requirements: Table Rock TX Tools

**Defined:** 2026-02-26
**Core Value:** Any new data pipeline — from source to Mineral — can be built by defining only the unique parts (source adapter + transform logic) while inheriting everything else (UI, ingestion, enrichment, export).

## v1.2 Requirements

Requirements for GHL API Integration milestone. Each maps to roadmap phases.

### API Configuration

- [x] **CONF-01**: User can add a GHL sub-account connection (name, Private Integration Token, Location ID) in Settings
- [x] **CONF-02**: User can manage multiple GHL sub-account connections (add, edit, delete)
- [x] **CONF-03**: Token + Location ID are validated together on save via test API call to GHL
- [x] **CONF-04**: Private Integration Token is stored encrypted in Firestore (never sent to frontend)

### Contact Management

- [x] **CTCT-01**: User can upsert contacts to GHL (create new, merge new info into existing)
- [x] **CTCT-02**: User can apply a campaign tag to all contacts in a batch
- [x] **CTCT-03**: User can optionally apply a "manual SMS" tag via checkbox
- [x] **CTCT-04**: User can assign a contact owner from a dropdown populated via GHL Users API
- [x] **CTCT-05**: System validates required fields (email or phone) before sending batch
- [x] **CTCT-06**: System handles GHL rate limits (100 req/10s) with automatic throttling and backoff

### Send UI

- [x] **SEND-01**: User sees "Send to GHL" button alongside "Download CSV" on GHL Prep results page
- [x] **SEND-02**: User sees a send modal with: sub-account selector, tag input, contact owner dropdown, SmartList/campaign name, manual SMS checkbox
- [x] **SEND-03**: User sees real-time progress bar during batch send showing contact count (X of Y)
- [x] **SEND-04**: User sees summary modal after send completion with created/updated/failed counts
- [x] **SEND-05**: User can download a CSV of failed contacts with error details

## Out of Scope

| Feature | Reason |
|---------|--------|
| Full GHL workflow automation | Triggers, workflows within GHL are managed in GHL UI, not from toolbox |
| Direct SmartList API creation | SmartLists are filter-based in GHL, not manual membership — use tags instead |
| Real-time sync on every edit | Burns rate limits, complex state management — batch send on user action |
| Automatic tag cleanup | Risk of removing existing GHL tags — additive only |
| Historical bulk import | Rate limits (200k/day) make full historical import impractical — focus on current data |
| OAuth marketplace integration | Private Integration Token is sufficient for internal use |
