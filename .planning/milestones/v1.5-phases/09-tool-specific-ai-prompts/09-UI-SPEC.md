## Component Changes

### ProposedChangesPanel.tsx — Confidence Badge in Group Header

**What changes:** Add a confidence badge to the collapsed group header row, showing the highest confidence level among changes in that group.

**Badge placement:** After the "Row {N}" text and before the "{N} fields" count badge. Insertion point is between the existing `<span className="text-sm text-gray-700">Row {entryIndex + 1}</span>` and the field count `<span>`.

**Badge markup:**
```
<span className="inline-flex items-center rounded-full px-1.5 py-0.5 text-xs font-medium {CONFIDENCE_STYLES[highestConfidence]}">
  {highestConfidence}
</span>
```

**Badge sizing:** `px-1.5` (6px horizontal), `py-0.5` (2px vertical), `text-xs` (12px), `font-medium` (500), `rounded-full`. Matches the existing field count badge dimensions exactly.

**Confidence aggregation logic:** For each group, compute the highest confidence among its changes using priority order: high > medium > low. Use a simple lookup map:
```
{ high: 0, medium: 1, low: 2 }
```
Select the confidence with the lowest numeric priority value.

**No changes to expanded detail view.** The per-change confidence pill already exists in the expanded detail (line 194 of current component). The new badge is for the collapsed/group header only.

### ProposedChangesPanel.tsx — Sort Groups by Confidence

**What changes:** Group entries are sorted so that groups with high-confidence changes appear first, then medium, then low.

**Sort point:** After the `groups` Map is built (line 56-65), before rendering. Convert to array, sort by highest confidence in group, then render.

**Sort order:** high first (priority 0), medium second (priority 1), low third (priority 2). Within same confidence tier, preserve original entry_index order (stable sort).

This phase has minimal frontend surface area. The vast majority of work is backend prompt authoring and pipeline plumbing (not covered by this UI spec). Frontend changes are limited to:

Items 2-4 are TypeScript plumbing with no visual changes. Only item 1 has UI contract implications (covered above).
